> **Chronology — read this in context.** This is an early, narrower audit dated 2026-07-14. It was
> superseded by six full adversarial rounds on 2026-07-19–20 that found **66 further confirmed

Adversarial review of the Sui-native contract (`async_lending.move`), the operator API, and the
web client, plus a live on-chain penetration test. Scope included the new features shipped this
session (dynamic rates, governance, per-collateral isolation).

**Headline: one CRITICAL vulnerability found and fixed; two hardening items; 8/8 live attacks blocked.**

---

## CRITICAL — collateral-type substitution (FOUND + FIXED)

**Where:** `disburse_attested<Collateral, Stable>`.

**The bug:** the operator's ed25519 attestation signed `borrower ‖ debt ‖ collateral_amount ‖
loan_commit` — it bound the collateral **amount but not its type**. The `Collateral` type is a
generic parameter chosen by the *caller*. So the on-chain check verified "the operator approved a
loan of N units of *something* for this debt" — not *which* asset.

**The exploit (real, direct fund loss):**
1. Attacker requests a legitimate quote for a valuable collateral — e.g. 1 cbBTC (1e8 units) →
   operator authorizes borrowing 4,000 USDC and signs the attestation.
2. Attacker deploys their own worthless coin `FAKE` (freely mintable) and mints 1e8 units.
3. Attacker calls `disburse_attested<FAKE, TUSDC>` with the FAKE coin + the *same attestation*.
4. `coin::value == 1e8` matches, the signature verifies (type wasn't signed), and they borrow
   **4,000 real USDC against worthless FAKE collateral.** Never repaid → bad debt → pool drained.

**The fix:** bind the collateral type into the signed message. The operator now signs the exact
collateral type; the contract reconstructs it in-Move via `type_name::get<Collateral>()` and appends
it to the verified message:
```
msg = bcs(borrower) ‖ bcs(debt) ‖ bcs(coll_amt) ‖ bcs(loan_commit) ‖ type_name(Collateral)
```
Now presenting a different coin type produces a different message → the signature fails →
`EBadAttest`. The off-chain SDK reconstructs the identical canonical type string
(`normalizeStructTag(coinType)` with the `0x` stripped), verified to byte-match `type_name` by a
**successful legit on-chain loop** (proving the fix doesn't break the real path).

**Verified:** pentest **P8** ("sig for SSPX, present TUSDC, same amount, everything else identical")
is now **BLOCKED** on-chain; SDK unit test `collateral-substitution is blocked`; legit type-bound
disburse still succeeds end-to-end through the hosted operator.

## Hardening (defense-in-depth)

- **`reserve_bps` unbounded** → a bad governance value `> 10000` would make the reserve skim exceed
  interest and could break accrual math. **Fixed:** `assert reserve_bps ≤ 10000` at init + on every
  `set_rate_curve`. (Governance-trusted, but bounding it removes a foot-gun.)
- **First-depositor share inflation (ERC-4626 class)** — known, pre-existing, MEDIUM. There is no
  public "donate to liquidity" function, so the only way to inflate share price without minting
  shares is accrued interest; and the live pool is seeded with a large first deposit (5,000 USDC),
  which makes the rounding attack impractical. **Mitigation deferred** (dead-share seeding) — tracked,
  not newly introduced.
- **Attestation replay** — an attestation has no on-chain single-use marker, so the same one could
  open multiple identical loans. LOW: each replayed loan is *fully collateralized with the now-type-
  bound collateral*, so no pool loss — it only lets a borrower take more identical safe loans than
  intended. Candidate for a used-`loan_commit` set later.

| P1 | forged attestation (random sig) | 🛡 BLOCKED | `EBadAttest` (ed25519) |

`set_rate_curve` / `set_collateral_cap`), utilization overflow (u128 intermediate), the P5 borrower
check on repay, and the on-chain proof gate on `settle_batch`.

## Honest takeaways
- The audit did its job: an adversarial pass on freshly-written code found a **real, critical,
  directly-exploitable** bug (undercollateralized borrow via type substitution) that unit tests +
  the happy path had missed — exactly the class of thing audits exist to catch.
- The formally-verified dregg kernel enforces the *LTV/solvency* invariant; but the **attestation
  bridge between the operator and the chain is ordinary code**, and that's where this bug lived. It's
  a reminder that "provably safe" covers the proven component, not the glue — the glue needs audits.
- Remaining known items (first-depositor inflation, attestation single-use) are documented, bounded,
  and not fund-loss on the live seeded pool; they're the honest backlog before mainnet.
