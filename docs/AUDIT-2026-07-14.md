# Platform audit + remediation — 2026-07-14

Two adversarial reviews (white-hat security + full UI/UX functional), findings verified
against the code, fixes applied and re-tested. Prior lending audit:
`cv-gateway/AUDIT-rwa-lending.md` (those items stay closed; these are NEW, mostly in the
pool + Operator API which postdate it).

## Security — CRITICAL / HIGH (all FIXED + verified)

| # | Sev | Finding | Fix | Verified |
|---|-----|---------|-----|----------|
| S1 | **CRITICAL** | **Pool vault not pinned** — deposit/withdraw read cash from a caller-supplied `pool_stable`; deposit into a fake empty vault inflates shares, then withdraw from the real vault. Net drain. | `require_pool_vault()` pins every vault to the pool authority's canonical ATA for (mint, token_program), in deposit/withdraw/disburse/repay/liquidate. `pool.rs`/`lib.rs`. | pool/async/liquidate flows still green (they use real ATAs); a non-canonical vault now errors `0xE2`. |
| S2 | **CRITICAL** | **Operator API trusted client-chosen feed/LTV/decimals** — a borrower could post junk collateral but claim `feedId=BTC, ltvBps=10000` → dregg authorizes a huge loan the operator co-signs. | Server-side `COLLATERAL_REGISTRY` keyed by mint → {feedId, ltvBps, decimals, tokenProgram}; the client's values are IGNORED; unknown mint → refused. `server.mjs`. | borrow E2E: `GUARD unknown-collateral: REJECTED ✓`; legit borrow still passes. |
| S3 | **HIGH** | Collateral vault unpinned (disburse/repay/liquidate) — same root cause as S1 for collateral. | Same `require_pool_vault` on `collateral_vault`. | flows green. |

## Security — MEDIUM / LOW
| Sev | Finding | Status |
|-----|---------|--------|
| MED | First-depositor / donation share-inflation griefing (ERC-4626 shape) | **Deferred** — the `shares==0` guard blocks the full theft (downgrades to rounding griefing). Full fix = lock dead shares at first deposit; it changes share accounting + needs test updates. Tracked. |
| MED | Operator API: CORS `*`, no auth, no rate-limit, `cargo run` per request (DoS) | **CORS locked** to the app origin(s). Auth + rate-limit + a warm long-running authorizer = prod-hardening, tracked. |
| LOW | `loan_commit` from `Math.random()` | **Fixed** — now `crypto.randomBytes(32)`. |
| LOW | Operator key on disk (`.operator.json`) | Note: 0600 / KMS in prod. Tracked. |
| LOW | No numeric validation on API amounts | **Fixed** — `posFinite()` on collateral/debt; body size capped. |
| LOW | share-count `u128→u64` truncation | Unreachable after S1; add a checked cast (tracked). |

*(The partial-sign flow was traced and is SAFE — the operator's signature covers the whole
message, so a client cannot alter the disburse after it's signed.)*

## Bottom line
**The fund-safety blockers (2 CRITICAL + 1 HIGH) are fixed and verified**, plus the two
borrow-path HIGH UX bugs and several quick wins. The deferred items are **hardening + polish**
(no fund-drain): dead-share seeding, API auth/rate-limit, positions health display, theme,
KPI ribbon, a11y, and the misc LOWs. None block the local demo; all should land before real
funds / mainnet.
